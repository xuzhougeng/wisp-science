# 手动项目同步

[English](project-sync.md)

Wisp 只会在你点击**立即同步**时同步项目，不会在后台定时同步，也没有文件监视器、WebSocket 或自动合并。同步期间项目会被独占锁定；如果项目中仍有对话任务、ACP turn、审批、审查、持久化写入或结构化运行，Wisp 会拒绝开始同步。同步完成前也不能在该项目中启动新任务。

每个同步修订包含：

- 一份完整但经过过滤的项目 SQLite 快照；
- 一份加密的工作区文件清单，路径统一使用 `/` 分隔的相对路径；
- 工作区中已变更文件的加密、内容寻址 blob。

项目元数据每次都会完整保存。未变化的工作区文件会复用之前的加密 blob，因此只上传发生变化的内容。同步后端只能看到项目与修订标识以及不透明的 blob 哈希，看不到项目名、对话、文件名、文件内容或项目加密密钥。正常拉取还会验证远端历史是否从本机最后已知的修订继续，防止修订描述被静默篡改或回滚。

## 选择同步后端

打开**设置 → 通用 → 手动项目同步**。

### 自托管中继服务器

将**存储后端**设为**自托管中继服务器**，然后填写 HTTPS 地址和 bearer token。Token 保存在操作系统密钥环中，不会写入项目数据或设备码。只有本地开发时的 `localhost` 地址可以使用明文 HTTP。

仓库内置的中继服务器可以这样启动：

```bash
export WISP_RELAY_TOKEN="replace-with-a-long-random-token"
export WISP_RELAY_ROOT="/var/lib/wisp-relay"
export WISP_RELAY_BIND="127.0.0.1:8787"
cargo run -p wisp-sync --bin wisp-relay --release
```

如果其他设备需要访问，请在中继服务器前配置 HTTPS 反向代理。服务器把文件保存在 `WISP_RELAY_ROOT`，通过原子替换写入，并在提交前比较客户端提供的基础修订和当前项目头。请像备份其他应用数据一样备份这个目录。

### 百度网盘、坚果云或其他共享目录

将**存储后端**设为**网盘共享目录**，再选择由网盘桌面客户端管理的本地目录。Wisp 会在其中创建 `Wisp Sync` 子目录，使用与中继服务器相同的加密修订与 blob 结构。Wisp 不会获得网盘服务商的 API 或账号凭据。

共享目录的绝对路径只属于当前设备。例如，Windows 可以使用 `D:\BaiduNetdisk\Wisp`，macOS 可以使用 `/Users/me/Nutstore/Wisp`。这些路径不会写入项目或设备码。

使用共享目录时：

1. 点击**立即同步**前，先等待网盘客户端下载完成。
2. 推送完成后，先等待网盘客户端上传完成，再切换到另一台设备。
3. 不要从两台设备同时同步同一个项目。

中继服务器能直接比较并交换远端修订，因此跨设备一致性强于普通网盘共享目录。

## 第一台设备

1. 结束或停止项目中的所有任务和运行。
2. 在项目卡片上点击**立即同步**。第一次同步会创建项目加密密钥和首个修订。
3. 点击**复制设备码**。

设备码包含项目加密密钥，请把它当作密码保管，不要发送给无关人员。中继服务器的 bearer token 不会包含在设备码里。

## 加入其他设备

1. 在新设备上配置中继 token，或选择该设备对应的本地网盘共享目录。
2. 打开**设置 → 通用 → 手动项目同步**，点击**加入同步项目**，粘贴第一台设备生成的设备码。
3. 点击**选择保存位置并加入**，选择一个新的本地父目录。

Wisp 会先下载到临时 staging 目录，验证每个加密 blob 和明文校验和，再把项目导入设备本地的工作区路径。加入完成后，设备码仍应继续按密码保管。

之后在一台设备上完成工作时，点击**立即同步**并等待中继或网盘客户端完成；再到下一台设备点击**立即同步**。没有本地修改的设备会拉取远端修订；有本地修改且基础修订仍是最新的设备会推送本地修订。

## 冲突与恢复

如果两台设备从同一个基础修订分别进行了修改，Wisp 会报告冲突，不会自动覆盖任何一方。冲突弹窗提供两个明确选择：

- **采用远端版本**：用远端当前修订替换本机的同步元数据和符合条件的本地文件。
- **采用此设备版本**：把本机完整状态发布为新修订，并把当前远端项目头作为它的父修订。

如果需要保留本机版本，请先导出项目 ZIP 作为安全备份，再处理冲突。修订和 blob 文件都是不可变的，因此即使选择了新的权威版本，之前的远端修订仍会保留在中继或共享目录中。

应用工作区文件时，Wisp 会在同一磁盘卷中使用 staging 和备份目录，并用一个小型恢复标记记录目标修订。如果应用在拉取期间退出，下次手动同步会根据 SQLite 是否已提交来完成清理或恢复之前的文件。

## 路径、排除项与限制

- 工作区路径使用相对路径和 `/` 分隔符；Windows 盘符和 macOS 绝对路径不会跨设备传递。
- 工作区内的 artifact 和 run 路径会重新绑定到目标工作区。工作区外的本机绝对路径会变成不可用引用，不会被猜测或改写到新设备。
- 运行时句柄、进程 ID、生命周期租约、远端工作目录、SSH 私钥路径、API 凭据、全局设置、SSH/WSL 配置和 ACP 进程绑定不会同步。
- 符号链接、特殊文件、Windows 不兼容名称以及大于 64 MiB 的单个文件只保留在本机，同步完成后会提示。大型科研数据应优先登记为带校验和与元数据的远端引用，而不是通过项目同步复制。
- 除非同步文件需要，空目录不会在另一台设备上创建。
- 两台设备应运行相同版本的 Wisp，以保持 protocol v1 快照兼容。
- Protocol v1 不会自动回收远端旧修订或 blob。删除本地项目只会移除本地同步密钥和游标，不会清除中继或网盘中的历史数据；需要时请单独删除后端数据。
- Protocol v1 不支持轮换项目密钥，也不能撤销已经复制的设备码。如果设备码泄露，请将项目迁移到新的同步存储位置，并删除旧后端历史。

手动 ZIP 导出/导入仍适用于离线迁移、归档备份，以及在处理冲突前保留本地副本。参见[项目导出与导入](project-transfer.md)。
